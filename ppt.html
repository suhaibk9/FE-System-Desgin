GraphQL

An Introduction to Typed, Queryable APIs

The Relational Challenge

In traditional REST, fetching nested relationships often requires multiple round-trips:

Scenario: Country Dashboard
  • GET /countries
  • GET /continents/{id}
  • GET /languages?countryId={id}
The Result:
  • Under-fetching: One call isn't enough.
  • Over-fetching: Getting 50 fields when you only need 2.
  • Waterfall: Client waits for Call A to start Call B.

The Schema Gap

REST relies on external documentation (Swagger/OpenAPI) which can drift from the actual code.

  • Lack of built-in type safety.
  • Frontend/Backend contract is often "implicit."
// Is this a string or object? "location": { "name": "United Kingdom", "code": "UK" }

The Solution: One Query

Instead of multiple endpoints, the client sends a "Shape" of the data it needs.

query {
  country(id: "UK") {
    name
    continent { name }
    languages { isoCode }
  }
}

Core Benefits

Strongly Typed

The Schema is the source of truth. Introspection allows for amazing tooling (GraphiQL).

Client-Driven

The frontend team can change the UI without asking the backend for a new endpoint.

A Balanced View: Trade-offs

  • Caching: More complex than REST (no native URL-based browser caching).
  • Query Complexity: Clients can write nested queries that strain the DB.
  • Learning Curve: Requires a shift in how you think about "Resources."
