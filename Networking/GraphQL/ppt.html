GraphQL

A query language for your API — and a simpler way to think about data fetching.

Context

Some Limitations of REST

REST works well — but as UIs get more complex, a few patterns start to feel limiting.

Over-fetching

The server returns a fixed shape. You needed 3 fields — you got 30. Extra data gets ignored.

Under-fetching

One request isn't enough. You hit /user, then /posts, then /comments — multiple round trips for one screen.

No enforced contract

The response shape is implicit — you find out at runtime if something changed on the server.

Endpoint growth

Every new UI variation often means a new endpoint or query param. APIs slowly sprawl over time.

Example 1 — Books & Authors

How many endpoints do you need?

Say you're building a site that shows Books, Authors, and their relationships.

With REST
  • GET /books
  • GET /authors
  • GET /books/:id/author
  • GET /authors/:id/books
  • GET /books?include=author
  • GET /authors?include=books
  • GET /books/:id?fields=title,author
With GraphQL — one endpoint, any shape

Enter GraphQL

        
One endpoint. You define the shape.

        

          With GraphQL the client asks for exactly the data it needs. The server
          resolves it — no more, no less.
        

        

          

            
Ask for exactly what you need

            

              The response shape mirrors your query — no wasted fields, no extra
              requests.
            

          

          

            
Single endpoint

            

              POST /graphql handles everything. No more endpoint sprawl across
              your codebase.
            

          

          

            
Schema as a contract

            

              Every field has a declared type. Queries are validated before they
              run — issues surface early.
            

          

          

            
Nested data in one request

            

              Books with their Authors with their other Books — resolved in a
              single query, not multiple round trips.
            

          

        

      


      
      

        

        
Comparison

        
REST and GraphQL, side by side

        

          
            

              Feature
              REST
              GraphQL
            

          
          
            

              Endpoints
              One per resource / operation
              Single endpoint
            

            

              Response shape
              Fixed by the server
              Defined by the client query
            

            

              Type safety
              Optional (OpenAPI / Swagger)
              Built-in schema
            

            

              Related data
              Multiple requests
              One nested query
            

            

              Versioning
              URL versions (v1, v2…)
              Deprecate fields, evolve schema
            

            

              Discoverability
              Manual docs / Swagger
              Auto-introspection, GraphiQL
